# ポモドーロタイマーと浄化タイマーの排他制御 設計書

関連 issue: [#87](https://github.com/TatsuhiroFuruta/get_the_time/issues/87)
過去の試行: [PR #88](https://github.com/TatsuhiroFuruta/get_the_time/pull/88)（未マージ）

## 目的

光の時間の活動（ポモドーロ計測 〜 活動記録の登録完了まで）と浄化タイマーを、同時に走らせられないようにする。あわせて、光の時間の活動が同時に 2 つ走ることも防ぐ。

## 守る範囲

**同一ブラウザの別タブまで。** 別ブラウザ・別端末からの同時利用は防がない。想定シナリオは「うっかり同じブラウザで 2 つ開いた」であり、悪意ある同時アクセスではない。

## 前提となる構造の非対称性

この機能を「1 つの仕組み」で作ろうとすると壊れる。2 つのタイマーは状態の在り処が違うためである。

- **浄化タイマーはサーバに状態を持つ。** `PurificationTime` が `status` / `started_at` / `total_time` を持ち、残り時間を `started_at` からの経過時間で導出する。タブを閉じても計測は続く。
- **ポモドーロはクライアントのメモリにしか状態がない。** サーバは活動記録が保存されるまで何も知らない。そして**信頼できる終了シグナルが存在しない**。ブラウザバック・タブ閉じ・記録せず離脱が、すべて「通知なき終了」になる。

PR #88 は `User#current_mode` という **明示的にクリアしなければ消えないフラグ**でこの状態を表そうとした。`disconnect()` から `/activity/cancel` を叩いて消す設計だったため、「離脱＝終了」と「フォームへの正常遷移」を JS で区別する必要が生まれ（`isNavigatingToForm`）、区別が効かない離脱（タブ閉じ・ブラウザバック）で破綻した。フラグが残ればユーザーは恒久的にロックアウトされる。

**本設計の核心は、フラグを「消す」のをやめ、「放っておけば腐って消える状態」で表すことである。**

## アーキテクチャ: 2 つの非対称なガード

### ガード A（サーバ側・新カラムなし）— 浄化タイマー計測中はポモドーロ画面に入れない

`ActivityRecordsController#pomodoro_timer` の `before_action` で判定し、マイページへリダイレクトする。浄化タイマーはタブを閉じても走り続けるため、**この向きを localStorage で判定してはならない**。サーバが唯一の真実である。

ただし既存の `PurificationTime#running?` をそのまま条件に使ってはいけない。`running?` は `status` カラムを見るだけで、時間切れの検知は JS がチャイムを鳴らした後に `stop!` を呼ぶことに依存している。**タブを閉じたまま時間切れになると `status` は `running` のまま残る。** これをロック条件にすると、PR #88 と同じ「消えないフラグ」をサーバ側で再現してしまう。

実時間から導出するメソッドを追加する。

```ruby
# 実時間ベースで「いま計測中か」を導出する。status だけを見ると、時間切れ後に
# stop! が呼ばれないまま（タブを閉じた等）running が残り、永久ロックになるため。
def counting?
  running? && started_at.present? && Time.current < started_at + total_time
end
```

`paused` はロック対象外とする。残り時間を保持して止まっているだけで、計測はしていない。

### ガード B（クライアント側・期限付きリース）— 光の時間の活動中は浄化タイマー画面に入れない

ポモドーロには終了シグナルがないので、**更新され続けている間だけ生きているリース**でロックを表す。

`localStorage` に置くのはこれだけ。

```json
{ "owner": "<タブID>", "expiresAt": 1720000000000 }
```

- **タブ ID は `sessionStorage`** に持つ。`sessionStorage` はタブ単位で、同一タブ内のページ遷移では保持され、別タブには引き継がれない。「タブの同一性」を表すのに正しい置き場所である。
- **ロック本体は `localStorage`**。タブ間で共有される唯一のストレージ。
- **TTL は 3 分、heartbeat は 30 秒間隔。**
- **期限切れのロックは「無い」ものとみなす。** 掃除役は不要。
- **ロックを削除する後始末は書かない。** 離脱の種類を区別するコードは一行も書かない。

これにより、ブラウザバック・タブ閉じ・記録せず離脱・ブラウザのクラッシュが、すべて「heartbeat が止まった」という一つの事象に畳まれる。特別扱いはゼロである。

そして PR #88 が転んだ箇所がここで解ける。**タイマー画面から活動記録フォームへの遷移は、同一タブなので `owner` が一致し、リースを更新し続けるだけで通り抜けられる。** `isNavigatingToForm` フラグも `/activity/cancel` エンドポイントも不要になる。

また `owner` が自分自身のロックは無視するため、同一タブで離脱して戻ってきたユーザーが自分のロックで締め出されることもない。

#### TTL が 3 分と長い理由（バックグラウンドタブのタイマー絞り）

ブラウザは**非表示タブの `setInterval` を絞る**。Chrome は 5 分ほど隠れたタブのタイマーを **1 分に 1 回**まで落とす（intensive throttling）。

これは他人事ではない。**ポモドーロ計測中に別タブへ切り替えるのは、まさにこの機能が想定している状況そのもの**である。TTL を数秒にすると、タブを離れて 5 分後にはリースがほぼ常時期限切れになり、浄化タイマーに入れてしまう。25 分のセッションのうち最初の 5 分しか排他が効かない機能になる。しかも Capybara のテストは数秒で終わるため、テストは全て緑のまま気づけない。

そこで、heartbeat が 1 分に 1 回まで絞られてもリースが切れない値（TTL 3 分 / heartbeat 30 秒）を選ぶ。

#### 離脱時の猶予（`pagehide` / `disconnect()` は削除ではなく期限の短縮）

TTL を 3 分にすると、離脱後もその間ロックが残ってしまう。そこで離脱時に自分のロックの**期限を 5 秒に詰める**（削除はしない）。

削除ではなく短縮にするのが要点である。**離脱の種類を判定しないまま、両方を正しく扱えるようになる。**

- 活動記録フォームへの正常遷移なら、遷移先のページが 1 秒以内に `renew()`（または再 `acquire()`）して満了期限が元に戻る。
- 本当の離脱（タブ閉じ・ブラウザバック）なら、誰も更新しないので 5 秒で腐って消える。

もし削除してしまうと、遷移中の一瞬だけロックが完全に空き、その隙に他タブが浄化タイマーへ入れてしまう。短縮ならその隙間は存在しない。そして PR #88 と違い、`isNavigatingToForm` のような「どちらの離脱か」を持つフラグは要らない。

**この短縮処理は 2 箇所から呼ぶ必要がある。**

- **`pagehide`**（`activity_lock.js` が自動で登録）— タブを閉じる・ブラウザバックする等、実際に document が破棄される離脱をカバーする。
- **Stimulus の `disconnect()`**（`activity_lock_controller.js` と `pomodoro_controller.js` それぞれで呼ぶ）— このアプリは Turbo Drive（`app/javascript/application.js` が `@hotwired/turbo-rails` を読み込む）を使っており、`link_to` によるアプリ内遷移は Turbo が横取りして DOM を差し替えるだけで、**document は作り直されない。つまり `pagehide` は一切発火しない。** 実際に活動記録フォームの「記録しない」（`mypage_path` への通常の `link_to`）をクリックした場合がこれに当たる。Stimulus の `disconnect()` は Turbo 遷移でも要素が DOM から外れる際に確実に呼ばれるので、ここでも `orphan()` を呼ぶ。`release()` ではなく `orphan()` である理由は `pagehide` と同じ：同一タブでフォームへ正常遷移する場合、遷移先の `connect()` が 1 秒以内に `acquire()` し直して満了期限を戻すため、削除ではなく短縮で足りるし、削除すると遷移中の一瞬だけロックが空く隙間ができてしまう。

ブラウザのクラッシュや強制終了では `pagehide` も `disconnect()` も発火せず、ロックは TTL の 3 分まで残る。これは稀なので許容する。

## コンポーネント

### `app/javascript/lib/activity_lock.js`（新規）

ロックの状態を読み書きするだけの純粋なモジュール。Stimulus に依存せず、画面のことも浄化タイマーのことも知らない。`localStorage` を触るコードをここに閉じ込め、キー名・TTL・「期限切れは無いものとみなす」というルールの解釈が 3 画面で食い違わないようにする。

| 関数 | 責務 |
|---|---|
| `acquire()` | 有効な他タブのロックが無ければ自分名義で取得。取得可否を返す |
| `renew()` | 自分が持ち主なら期限を延ばす。持ち主でなければ何もしない |
| `release()` | 自分が持ち主なら削除 |
| `heldByOther()` | 他タブが有効なロックを持っているか（期限切れは false） |
| `orphan()` | 自分が持ち主なら期限を 5 秒に詰める。`pagehide` で自動的に呼ばれるほか、`activity_lock_controller` / `pomodoro_controller` の `disconnect()` からも呼ぶ（Turbo Drive の遷移では `pagehide` が発火しないため） |

### `app/javascript/controllers/activity_lock_controller.js`（新規）

上のモジュールを画面に貼るための薄い Stimulus コントローラ。値は 2 つ。

- `hold: Boolean` — true なら connect 時にロックを取り、`HEARTBEAT_MS`（30 秒）ごとに `renew()` し続ける。
- `redirectTo: String` — `heldByOther()` なら、この URL へ `location.replace()` する（`guard()` メソッド）。

`localStorage` の読み取りは同期なので、判定とリダイレクトは connect と同じティックで走る。画面のちらつきはほぼ無い。

**ただし connect 時の判定だけでは足りない。** 両方のタブを「ロックが無い」状態のまま先に開いておき、その後で片方をスタートする、という後出しの操作は connect 時のガードをすり抜ける。これを塞ぐため `guard()` を次の 2 つのイベントにも束ねる（`disconnect()` で両方とも解除する）。

- **`storage` イベント** — 他タブが `localStorage` を書き換えた瞬間に、その書き換えを行っていない側のタブだけで発火する。開きっぱなしの浄化タイマー画面が、別タブでポモドーロがスタートされた瞬間に自分から `guard()` を再実行して追い返される。
- **`pageshow` イベント** — bfcache（ブラウザの戻る/進むによる復元）をカバーする。bfcache からの復元では Stimulus の `connect()` が再実行されないため、これが無いと復元後にガードが働かない。

これでも塞げないのは、浄化タイマーの計測中判定（ガード A）で、こちらはサーバに問い合わせないと分からない。そのためのスタート押下時ガードは以下で別途説明する。

### 各画面への割り当て

| 画面 | ガード（他タブが持っていたら弾く） | 保持（heartbeat） |
|---|---|---|
| `purification_times#show` | あり → マイページへ | なし |
| `activity_records#pomodoro_timer` | あり → マイページへ | **スタート押下時**に `pomodoro_controller` が取得 |
| `activity_records#new`（記録フォーム） | なし | あり（`hold: true`） |

ポモドーロ画面だけ保持のトリガーが特殊（画面表示ではなくスタート押下）なので、そこは `pomodoro_controller` 自身が `acquire()` / `renew()` / `release()` を呼ぶ。**タイマー画面を開いて眺めているだけではロックしない。** フォーム画面は表示している間ずっと保持で良いので、汎用の `activity_lock_controller` に `hold: true` を付けるだけでよい。

ポモドーロ画面にもガードを付けることで、issue #87 の 3 項目目「光の時間の活動は 1 つしか行えない」も同じ仕組みで満たされる。

### スタート押下時のガード（画面を開いた後出しの操作への対策）

connect 時（＋ `storage` / `pageshow`）のガードは「先に開かれていた画面」を追い返せるが、**両方のタブを両方とも「ロックが無い」状態のまま先に開いておき、その後で片方をスタートする**という順序には無力な組み合わせが 2 つ残る。

- 浄化タイマー画面を開いたまま（まだロックが無いのでガードが通る）→ 別タブでポモドーロがスタートされる → 浄化タイマー画面で「スタート」を押す。
- ポモドーロ画面を開いたまま（浄化タイマーはまだ計測中ではないのでガードが通る）→ 別タブで浄化タイマーがスタートされる → ポモドーロ画面で「スタート」を押す。

どちらも「画面を開いた時点」ではなく「スタートを押した時点」でもう一度判定しないと防げない。そこで両方の `start()` に、実際にサーバへ状態を変更する前のチェックを追加した。

- **`purification_timer_controller.js#start()`** — PATCH `/purification_time/start` を送る前に `heldByOther()` を確認する。他タブが光の時間の活動ロックを持っていれば `location.replace("/mypage?locked=activity")` して PATCH 自体を送らない。
- **`pomodoro_controller.js#start()`** — 初回スタート時のみ、タイマーを起動する前に `GET /purification_time`（`Accept: application/json`）に問い合わせる。`PurificationTimesController#show` の JSON レスポンスに `counting`（`PurificationTime#counting?` の値）を追加してあり、`true` なら `location.replace("/mypage?locked=purification")` してタイマーを起動しない。通信に失敗した場合は排他制御より本来の機能を優先し、計測中ではないものとして扱う。

浄化タイマー側はロックの所在が `localStorage` にあるので同期的に判定できるが、ポモドーロ側は判定材料（`counting?`）がサーバにしかないため、非同期の問い合わせが必要になる。`start()` を `async` にしたのはこのためである。ロック取得 → スタートボタンを隠す、という既存の順序はこのチェックの後に維持している。

なお `PurificationTimesController#show` は `has_one :purification_time` によりレコードが存在しないユーザー（浄化タイマーを一度も付与されていない）もあり得るため、JSON 分岐は `@purification_time&.running? || false` / `@purification_time&.counting? || false` と nil セーフにしてある。ポモドーロ側の初回スタートは全ユーザーが通る経路なので、ここが nil で 500 すると排他制御を追加したことでポモドーロ自体が使えなくなってしまう。

`mypages_controller.rb` は `params[:locked] == "purification"` のときに `mypages.flash_message.purification_counting`（「浄化タイマーの実行中はポモドーロタイマーを開始できません」）を表示する。`"activity"` のときの既存メッセージ（`mypages.flash_message.activity_locked`）と同じ仕組みである。

## 明示的に解放する場面

TTL 待ちを避けたい 1 箇所だけ。

- **記録フォームの submit 時** → `release()`。バリデーションエラーで `new` が再描画された場合は、そのとき `acquire()` し直すので自己修復する。

これ以外で明示的に削除することはない。離脱系はすべて `pagehide` による期限短縮と TTL 任せである。

## エラーハンドリングと既知の限界

- **`localStorage` が使えない環境**（プライベートモード等で例外を投げる場合）では、`activity_lock.js` の全関数が例外を飲み込み、「ロックは無い」ものとして振る舞う。排他制御が効かなくなるだけで、アプリは通常どおり動く。**排他制御のために本来の機能を壊さない**方針。
- **フォーム submit 直後の数百 ms**、ロックが解放された状態でサーバが処理している。この隙に他タブが浄化タイマーを開始できる。ポモドーロの計測は既に終わっているため実害は小さく、これ以上の防御はしない。
- **クライアント側ガードは JS 前提。** 浄化タイマーの `start` を PATCH で直接叩けば、スタート押下時ガード（`heldByOther()` チェック）は素通りできる。逆向き（ガード A）はサーバで守られており非対称だが、それは状態の在り処が非対称だからである。MVP としてはこれで十分とする。
- **別ブラウザ・別端末**は防がない（スコープ外）。
- **完全にバックグラウンド化されたタブは、Chrome にタイマーごと「凍結（freeze）」されることがある。** 「絞られる（throttling、1 分に 1 回まで低下）」に留まらず、`setInterval` が完全に停止するケースが Chrome にはある。この場合 heartbeat も止まるため、ポモドーロの計測自体は継続していても、ロックは TTL の 3 分でそのまま腐って消える。以降そのポモドーロは他タブからの排他対象ではなくなる。頻度・実害ともに小さく、許容する（凍結を検知して能動的に何かする仕組みは持たない）。
- **フォーム送信後にブラウザバックでキャッシュされた記録フォームへ戻ると、その画面の `activity-lock` コントローラが `connect()` で再度 `acquire()` し、既に登録済みの記録に対して新しい 3 分のロックを取り直してしまう。** 誰も `renew()` しなければ TTL 相当（実質は離脱時の `orphan()` によりもっと早く）で腐って消えるので、他タブの浄化タイマーが一時的にブロックされるだけで、放置すれば自己修復する。恒久的なロックアウトにはならないため、これ以上の防御はしない。

## スコープ外

- ポモドーロ計測中のブラウザバックそのものの抑止（`history.pushState` 等）。これは排他制御ではなく「計測データを失わせない」という別の課題であり、別 issue とする。既存の `beforeunload` 確認ダイアログは今のまま維持する。
- 活動記録の登録後、ブラウザバックでフォームに戻れてしまう二重登録の可能性（既存の挙動であり、本 issue とは別問題）。

## テスト方針

- **モデル spec**: `PurificationTime#counting?`。特に「`running` のまま時間切れになったら false を返す」ことを `travel_to` で検証する。ここが永久ロックを防ぐ要である。
- **リクエスト spec**: 浄化タイマーが計測中のとき `GET /activity_records/pomodoro_timer` がマイページへリダイレクトすること。時間切れの `running` では通ること。
- **システム spec**: Capybara の `open_new_window` / `within_window` で 2 タブを開き、ロックが別タブへ効くことを検証する。`localStorage` は同一ブラウザセッションで共有されるため実際に再現できる。
- JS のユニットテスト基盤は本プロジェクトに無いため、`activity_lock.js` はシステム spec 経由で検証する。
